# Release

Bumps the version, updates the changelog, creates a git tag, and pushes to trigger the CI release pipeline. Optionally runs deploy steps documented in `docs/workflow/deploy.md`.

## Usage
```
/release
/release patch
/release minor
/release major
```

## Instructions

### 1. Pre-flight Checks
- Verify current branch is the release branch (usually `main` or `master`)
- Run `git status` — working directory must be clean. If not: stop and report uncommitted changes.
- Run all tests. Stop if any fail.
- Read `docs/workflow/release.md` to understand the project's release configuration.

### 2. Determine Bump Type
If no bump type was provided as an argument, ask the user:
```
What type of release?
- patch (bug fixes only, no new features)
- minor (new features, backwards-compatible)
- major (breaking changes)
```

### 3. Calculate New Version
Read the current version from the project's source of truth:
- TypeScript/Node: `node -p "require('./package.json').version"`
- Python: `grep -m1 '^version' pyproject.toml | cut -d'"' -f2`
- Rust: `grep -m1 '^version' Cargo.toml | cut -d'"' -f2`
- C++: extract from `CMakeLists.txt` `project(... VERSION x.y.z ...)`

Apply semver bump (patch: increment z, minor: increment y reset z, major: increment x reset y.z).

### 4. Update Version in Source of Truth
- TypeScript: use `npm version {bump_type} --no-git-tag-version`
- Python: update `version = "x.y.z"` in `pyproject.toml`
- Rust: update `version = "x.y.z"` in `Cargo.toml`
- C++: update `project(... VERSION x.y.z)` in `CMakeLists.txt`

### 5. Update Build-Time Version Injection
If the project has a version injection file (e.g., `src/version.ts`, `src/version.py`):
- Generate/update it with the new version and current git commit hash:
```typescript
// Auto-generated — do not edit
export const VERSION = "{new_version}";
export const GIT_COMMIT = "{git rev-parse --short HEAD}";
```
```python
# Auto-generated — do not edit
VERSION = "{new_version}"
GIT_COMMIT = "{git rev-parse --short HEAD}"
```
```
/* Auto-generated by CMake — do not edit */
```
(For C++ the cmake configure_file handles it; no manual update needed.)

### 6. Update CHANGELOG.md
Get commits since the last tag:
```
git log $(git describe --tags --abbrev=0)..HEAD --oneline --no-merges
```

Group by conventional commit type:
```markdown
## [1.2.3] - 2026-06-10

### Features
- feat(auth): add OAuth login with Google and GitHub
- feat(api): add pagination to list endpoints

### Bug Fixes
- fix(session): correct token expiry calculation

### Internal
- chore: update dependencies
- refactor(auth): extract OAuthProvider interface
```

Insert this entry at the top of `CHANGELOG.md` (after the `# Changelog` header).

### 7. Commit, Tag, Push
```
git add -A
git commit -m "chore: release v{new_version}"
git tag v{new_version}
git push
git push --tags
```

### 8. Wait for CI Release
If GitHub remote exists: monitor CI pipeline:
```
gh run list --workflow=release.yml --limit=1
```
Print the URL and report status. If it fails: report and stop.

### 9. Deploy (optional)
If `docs/workflow/deploy.md` exists and describes automated deploy steps that /release should trigger:
- Read the deploy documentation
- Execute the documented deploy steps
- If a step requires user confirmation or credentials: pause and ask

If deploy is manual (e.g., SSH to server): print the steps from deploy.md for the user to follow.

### 10. Report
```
Released v{new_version} ✓
Tag: v{new_version}
Changelog: updated
CI: {pass/running/url}
{If deploy: Deploy: {status}}

Full release notes: https://github.com/{repo}/releases/tag/v{new_version}
```
