# Release

Bumps the version, updates the changelog, creates a git tag, and pushes to trigger the CI release pipeline. Optionally runs deploy steps documented in `docs/workflow/deploy.md`.

## Usage
```
/release
/release patch
/release minor
/release major
```

## Instructions

### 1. Pre-flight Checks
- Read `docs/workflow/release.md` to understand the project's release configuration, especially the **branching model**. If the file doesn't exist (project onboarded with an older plugin version): continue without it and offer to create it from `templates/workflow/release.md.template` afterwards.
- Determine the branching model:
  - **git flow** if release.md says so, or if a `develop` branch exists
  - **main-only** otherwise
- Verify you are on the correct branch:
  - main-only: the release branch (`main` / `master`)
  - git flow: `develop` (the release is prepared on develop, then merged to `master`)
- Run `git status` — working directory must be clean. If not: stop and report uncommitted changes.
- Pull latest (`git pull`) so the release includes everything merged so far.
- Run all tests and the linter via the `test-runner` subagent (keeps the output out of the main context). Stop if anything fails — in git flow, fix bugs on develop (via the normal feature/fix workflow or direct commits for trivial fixes) before releasing.

### 2. Determine Bump Type
If no bump type was provided as an argument, ask the user:
```
What type of release?
- patch (bug fixes only, no new features)
- minor (new features, backwards-compatible)
- major (breaking changes)
```

### 3. Calculate New Version
Read the current version from the project's source of truth:
- TypeScript/Node: `node -p "require('./package.json').version"`
- Python: `grep -m1 '^version' pyproject.toml | cut -d'"' -f2`
- Rust: `grep -m1 '^version' Cargo.toml | cut -d'"' -f2`
- C++: extract from `CMakeLists.txt` `project(... VERSION x.y.z ...)`

Apply semver bump (patch: increment z, minor: increment y reset z, major: increment x reset y.z).

### 4. Update Version in Source of Truth
- TypeScript: use `npm version {bump_type} --no-git-tag-version`
- Python: update `version = "x.y.z"` in `pyproject.toml`
- Rust: update `version = "x.y.z"` in `Cargo.toml`
- C++: update `project(... VERSION x.y.z)` in `CMakeLists.txt`

### 5. Build-Time Version Injection
The version source of truth is the manifest bumped in step 4 (`package.json` / `pyproject.toml` / `Cargo.toml` / `CMakeLists.txt`). Build-time version files like `src/version.ts` or `src/version.py` are **generated by the build** (e.g. `scripts/generate-version.js` runs during `npm run build`) and are typically **gitignored** — do NOT hand-edit or commit them.

- **If the version file is gitignored** (the default for the TypeScript template): do nothing here. It is regenerated on the next build.
- **If the project commits its version file** (no gitignore entry, no build-time generator): update it manually so the committed value matches the new version:
```typescript
// Auto-generated — do not edit
export const VERSION = "{new_version}";
export const GIT_COMMIT = "{git rev-parse --short HEAD}";
```
```python
# Auto-generated — do not edit
VERSION = "{new_version}"
GIT_COMMIT = "{git rev-parse --short HEAD}"
```
(For C++ the cmake `configure_file` handles this at build time; no manual update.)

Check `.gitignore` before editing: if the file is ignored, skip it.

### 6. Update CHANGELOG.md
Get commits since the last tag:
```
git log $(git describe --tags --abbrev=0)..HEAD --oneline --no-merges
```

Group by conventional commit type:
```markdown
## [1.2.3] - 2026-06-10

### Features
- feat(auth): add OAuth login with Google and GitHub
- feat(api): add pagination to list endpoints

### Bug Fixes
- fix(session): correct token expiry calculation

### Internal
- chore: update dependencies
- refactor(auth): extract OAuthProvider interface
```

Insert this entry at the top of `CHANGELOG.md` (after the `# Changelog` header).

### 7. Commit, Tag, Push

**Main-only model:**
```
git add -A
git commit -m "chore: release v{new_version}"
git tag v{new_version}
git push
git push --tags
```

**Git flow model** (the tip of `master` must always equal the latest released state):
```
# 1. Release commit on develop
git add -A
git commit -m "chore: release v{new_version}"
git push origin develop

# 2. Merge develop into master and tag the merge commit
git checkout master
git pull origin master
git merge --no-ff develop -m "chore: merge develop for release v{new_version}"
git tag v{new_version}
git push origin master --tags

# 3. Sync the merge commit back into develop, return to develop
git checkout develop
git merge master
git push origin develop
```
If the merge into `master` produces conflicts: stop and ask the user (in unsupervised mode: write `## Blocked`).

### 8. Wait for CI Release
If GitHub remote exists: monitor CI pipeline:
```
gh run list --workflow=release.yml --limit=1
```
Print the URL and report status. If it fails: report and stop.

### 9. Deploy (optional)
If `docs/workflow/deploy.md` exists and describes automated deploy steps that /release should trigger:
- Read the deploy documentation
- Execute the documented deploy steps
- If a step requires user confirmation or credentials: pause and ask

If deploy is manual (e.g., SSH to server): print the steps from deploy.md for the user to follow.

### 10. Report
```
Released v{new_version} ✓
Tag: v{new_version}
Changelog: updated
CI: {pass/running/url}
{If deploy: Deploy: {status}}

Full release notes: https://github.com/{repo}/releases/tag/v{new_version}
```
